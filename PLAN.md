Build the actual tactical RPG game on top of the reusable engine library.

### Folder Roles (Why Similar Names Exist)

- `game/src/`: implementation code (`.cpp`) and game-private headers (`.h`) used directly by the game target.
- `game/assets/`: runtime content (maps, sprites, audio, units, skills) copied next to the executable.
- `build/`: generated files only (Visual Studio projects, object files, caches). Safe to regenerate.
- `tools/`: scripts that configure/build with the right environment.

### Include vs Src Rule of Thumb

- Put declarations in headers (`.h`) and definitions in sources (`.cpp`).
- If a header is only for this game executable, keep it under `game/src/...`.
- If a stable public game API is needed later, add `game/include/...` at that time.

### Include Style Convention (Game Source)

- Keep includes rooted from `game/src` include dirs, for example: `states/BattleState.h`, `battle/BattleMap.h`, `ui/DialogBox.h`.
- Do not use workspace-absolute style include strings like `game_1/src/...`.
- In each `.cpp`, include order should be:
  1. Matching module header first (for example `states/BattleState.h`).
  2. Other game headers (`states/...`, `battle/...`, `ui/...`, `data/...`, `ai/...`).
  3. Engine headers (`engine/...`).
  4. Third-party headers (`SDL3/...`, `nlohmann/...`, etc.).
  5. C++ standard library headers (`<vector>`, `<string>`, etc.).
- Keep include paths and casing consistent to avoid platform/tooling issues.

### About "Same Filename in Different Paths"

- `X.h` and `X.cpp` with the same base name are expected and correct: header declares, source implements.
- Similar paths across `engine/` and `game_1/` are also expected by design:
  - `engine/include/...` and `engine/src/...` are engine API vs engine implementation.
  - `game_1/game/src/...` contains game-specific logic built on top of engine APIs.
- Avoid duplicating two different headers with the same include path contract across folders.

## Phases

### G1 — Bootstrap & Integration

1. Define game executable target and link to `engine`
2. Keep dependencies aligned in `vcpkg.json`
3. Ensure startup path from `main.cpp` to first state works

### G2 — State Flow

4. `BootState`: preload minimal resources and transition
5. `MainMenuState`: input-driven transition to gameplay
6. `BattleState`: own game loop context
7. `PauseState`, `SettingsState`, and `ResultState`: temporary and terminal flow

### G3 — Grid, Map, and Navigation

8. `Grid` data model with tile metadata
9. Consume map data via engine `TiledJsonLoader` and project gameplay metadata from layers/properties
10. Cursor movement and tile selection UX
11. `Pathfinder` (A\*) and `MovementRange` (BFS)

### G4 — Units, Turns, and Combat

12. `UnitData` + runtime `Unit`
13. `TurnQueue` initiative order
14. `CombatSystem::resolve()` with hit/crit/damage logic
15. KO handling and win/lose conditions

### G5 — Enemy AI

16. `EnemyAI::takeTurn()` heuristic movement/targeting
17. Integrate AI turn into battle loop

### G6 — Player UI

18. `UnitPanel`, `ActionMenu`, `DamagePreview`
19. Turn indicator and round feedback
20. Dialog flow integration (`DialogBox`, `DialogSystem`) built on engine UI primitives

### G7 — Data & Content Pipeline

21. JSON schemas for maps, units, skills
22. `UnitLoader` and `SkillLoader`
23. Optional Debug hot-reload for JSON content

### G8 — Polish

24. Replace placeholder unit rendering with real sprites/animations
25. Add SFX and BGM
26. Add presentation polish (screen shake/highlights/effects)
