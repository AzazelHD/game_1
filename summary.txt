New this session — Settings refactor & menu inconsistency

Settings split into three states + data‑driven architecture
The monolithic SettingsState has been fully decomposed into:

    SettingsState (hub) — three‑item menu (Graphics / Audio / Back), data‑driven using a RowItem vector with lambdas for actions.

    GraphicsSettings — resolution & window mode, with its own confirmation popup for unsaved changes.

    AudioSettings — master & music volume sliders, with a confirmation popup for unsaved volume changes (new feature; previously audio changes were discarded silently).

All three use a shared SettingsManager singleton (consistent with the rest of the engine’s FontManager, Input, etc.) to load/save/apply settings from disk, and a UIUtils helper namespace containing drawRow and formatButtonLabel that were previously duplicated across the old code.

New/updated files:

    data/SettingsManager.h/.cpp — singleton replacing the old SettingsData struct, holds the single copy of settings.

    ui/UIUtils.h — centralized drawRow and formatButtonLabel (with optional prefix/suffix parameters).

    states/SettingsState.h/.cpp — refactored hub.

    states/GraphicsSettings.h/.cpp — refactored graphics page.

    states/AudioSettings.h/.cpp — refactored audio page.

No changes to MainMenuState or WorldMapState — they still push SettingsState with the same two‑argument constructor, because the settings manager is a global singleton.

Old audio slider fix superseded — the previous patch that removed m_editingSlider is no longer relevant; the refactor naturally makes sliders directly adjustable with Left/Right/A/D and applies changes immediately.

Menu inconsistency identified & marked as priority
The new settings screens use custom manual row rendering (drawing rectangles, positioning text) while MainMenuState uses the UIManager + ActionMenuWindow widget system. This is inconsistent — the project now has two different ways of building menus. The desired state is:

    All menus should work the same way: a single UIManager‑driven window system, with reusable window types for settings rows (including sliders and buttons).

Priority: High — this should be resolved before the settings code grows further. The plan would be to create a SettingsWindow widget (or extend ActionMenuWindow) so that all three settings states become thin wrappers that just push a window, just like MainMenuState does. This would also make it trivial to add new settings pages or repurpose the same window elsewhere.
