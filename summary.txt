Deployment auto-advance-selection bug fixed — root cause: placeGrabbed() blindly advanced m_selectedIndex by +1 regardless of whether that landed on an already-placed entry, causing an intermittent "cursor locks every other re-place" bug (confirmed by user's A/B parity test). Fixed to search forward for the next unplaced entry instead of a blind +1.
Selection now follows what's in hand — DeploymentSystem::grabUnit() updates m_selectedIndex to match whatever was just grabbed (fresh from roster or re-picked from a placed tile) — fixes Tab/Details showing stale roster selection instead of the actually-held unit. processUIEvents's ActionId::Details handler reordered to prioritize grabbedEntry() first, then hover, then roster selection.
Cycling with a unit in hand now swaps hand contents — DeploymentSystem::cycleSelection() no longer blocks while a unit is grabbed; instead, Q/E now drops whatever's held (returns to roster unplaced, regardless of whether it was placed before being grabbed — explicit user decision, equivalent to ESC→cycle→Enter as one step) and grabs the newly-selected entry instead.
Swap-in-place implemented — new DeploymentSystem::swapGrabbedWithPlacedAt(pos): Enter with a unit in hand, over a tile occupied by another of the player's own placed units, now swaps them (occupant unplaced and becomes the new held unit; held unit placed into that tile) instead of blocking as "occupied." Confirmed scoped to the player's own roster only — deployedEntryAt structurally cannot return enemy/ally/neutral preview units (those live in a separate list DeploymentSystem never touches).
Roster identity redesigned (major) — RosterUnit/DeploymentEntry moved off std::string unitId (was literally "soldier"/"aria" — the class/job template name, not a unique identity; would collide if recruiting multiple Soldiers) onto int instanceId, a monotonically-increasing counter assigned once at recruitment, never reused even after dismissal (rejected linked-list/FIFO alternative — overflow isn't a real concern at int range, and reusing ids risks silent identity confusion with any stale reference, e.g. saved events/logs). RosterUnit also gained customName (for future unique named units — Marche, Montblanc) and isCritical (wires into BattleSession::m_criticalUnits/DefeatConditionType::CriticalUnitDied, built earlier but never actually populated until now — Aria is recruited with isCritical=true). Full migration completed across RosterSystem.h/.cpp, PartySystem.h/.cpp (memberIds() now returns std::vector<int>), DeploymentSystem.h/.cpp (every unitId comparison → instanceId), and BattleState.cpp (deployment window labels and unit-panel preview fallback now load the real display name from the unit's template via UnitLoader::load instead of ever showing a raw id).
Placeholder unit visuals (circle + letter) added, replacing debug-only circles — new UnitPortrait::drawPlaceholderSprite(renderer, font, center, diameter, team, letter, alpha): draws a team-colored filled circle with a centered initial-letter glyph, used as a stand-in until real unit sprites exist. Replaces DebugRenderer::addScreenCircle calls in BattleRenderer::drawUnitAt (map) and BattleState::render()'s grabbed-unit indicator (Deployment); also wired into UnitPortrait::renderFromStats's image slot as a fallback when no real sprite file is found (was previously silently blank — the placeholder branch was originally nested inside if(hasImage) and never reached when no path resolved at all; fixed to always reserve the image slot and draw something, totalW also fixed to always include image width). Letter is always derived from the unit's real display name (getName()[0] / loaded from template) — never from instanceId or the old unitId, avoiding the lowercase/uppercase mismatch bug found when the map used getName() but the hand used the (now-removed) string id.
Alpha param uses uint8_t (0-255), not float (0-1) — matches the existing Color struct's own channel convention, per user's explicit preference; also confirms why team stays int (not uint8_t): team is a category/identity value including a meaningful negative (-1 = neutral in teamFrameColor), not a 0-255 intensity — uint8_t would silently wrap -1 to 255 and break the neutral case.
FontManager introduced (major, engine-level) — new engine/renderer/FontManager.h/.cpp, a singleton owning every Font* keyed by FontRole (Body, Heading, Title, Placeholder), loaded once via loadAll(Renderer*) from a single table (path+size per role) instead of scattered per-state loadFont calls. Replaces App's old s_defaultFont/getDefaultFont()/setDefaultFont() entirely (removed from App.h/.cpp; App::~App() now calls FontManager::instance().clear()). BootState::onEnter() now just calls FontManager::instance().loadAll(m_renderer) instead of manually loading one font. All prior App::getDefaultFont() call sites need migrating to FontManager::instance().get(FontRole::Body) (or ::Heading/::Title/::Placeholder for the few special cases — main menu title, placeholder sprites) — user is doing this migration manually across the codebase; not yet confirmed complete.
